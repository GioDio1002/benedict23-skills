Metrics & Data Pipeline

Precise definitions, formulas, and on-court meaning for the ten radar axes used across the per-team decks and the comparison page, plus the full data acquisition pipeline behind them.

Every originally "lower is better" raw metric is published as its positive opposite (Ball Security / Def. Stops / Foul Discipline / Matchup Suppression), so "further from center = better" holds across the radar. The four defensive additions — Block %, Steal %, Forced TOV %, Matchup Suppression — are queued for the next re-scrape that expands the radar to fourteen axes.

Data Acquisition Pipeline

1 · Source

Official `stats.nba.com` data only, via `nba_api` (`LeagueGameFinder`, `TeamGameLogs`, `BoxScoreTraditionalV3`, `BoxScoreAdvancedV3`) covering all six 2026 Finals games plus each team's full playoff run.

2 · Strategy

Pull game lists by team ID × season ID × SeasonType=Playoffs, then per-game traditional + advanced box scores by game ID. Raw JSON is cached under `data/raw//`.

3 · Polite scraping

`stats.nba.com` rate-limits aggressively: 0.6–1.2s jittered sleep between calls + exponential backoff on failure, project-identifying UA. All analysis reads from cache — no double-hits.

4 · Three baselines

For every player we build three series: (i) this Finals series, game-by-game aggregated; (ii) the player's full playoff average (Finals included); (iii) team-wide Finals average. The radar plots all three for a player / self-baseline / team-baseline triangle.

5 · Derived metrics

Score Share, Ball Security (100−TOV%), Def. Stops (120−DefRtg), Foul Discipline (6−PF/G) are computed explicitly before normalization. Formulas live in `derive.py` so they stay auditable.

6 · Verification

Cross-check aggregated team totals + W/L per game via `BoxScoreSummaryV2`. Any per-player TS% / USG% diverging >0.5pp from the public stats.nba.com page triggers a re-pull.

7 · Normalization

Radar drawing uses per-axis min/max normalization across the page's player set, so any two players are area-comparable; the underlying tables still report raw values.

8 · Defensive matchup data (expansion)

The four defensive expansion axes come from additional endpoints: BLK% / STL% read directly from BoxScoreAdvancedV3; Forced TOV% attributes each TOV (EVENTMSGTYPE=5 with SCREEN_AST / STEAL chains) to the defender via PlayByPlayV3; Matchup FG% Allowed pulls per-matchup-possession off/def player pairs from LeagueDashPtDefend or BoxScoreMatchupsV3 and aggregates FGM/FGA by DEF_PLAYER, then publishes 100 − MFG% so higher = better matchup defense.

All scripts and intermediate artifacts live under src/skills/nba-finals-radar-publishing/ for independent reproduction.
